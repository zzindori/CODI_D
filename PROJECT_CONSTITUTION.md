# 프로젝트 개발 헌법 (v1.2)

## [0. 헌법의 지위]

이 문서는 본 프로젝트의 최상위 설계 헌법이다.
이후 등장하는 모든 프롬프트, 기능 요청, 예시, 코드 제안보다
항상 우선 적용된다.

본 헌법을 위반한 설계·구현은
오류로 간주하며 수정 후 다시 제안해야 한다.

이 지침은 본 프로젝트 전체 대화에 공통 적용된다.

────────────────────────────────

## [1. 프로젝트 성격]

본 프로젝트는
사용자의 단 한 번의 행동(one-shot, one action)이
곧바로 결과 화면으로 이어지는
**이벤트 중심 앱 흐름**을 설계·정리·구현하기 위한 프로젝트다.

파일 업로드/다운로드 중심 구조가 아니라,
이벤트 발생 → 즉시 결과 반영 구조를 기본 전제로 한다.

────────────────────────────────

## [2. 공통 설계 철학]

- 사용자의 행동 1번은 즉시 결과로 이어져야 한다
- 저장은 선택이며, 자동 처리가 가능한 흐름을 우선한다
- 기능 추가보다 '중간 사고 단계를 줄이는 UX'를 우선한다
- 앱은 도구가 아니라 '결정을 빠르게 끝내주는 흐름'이어야 한다

────────────────────────────────

## [3. 설계 기준]

- 기능 구현 이전에 사용자 흐름과 화면 구조를 먼저 정의한다
- MVP 범위와 이후 확장 범위를 명확히 구분한다
- 이벤트 발생과 결과 반영의 연결 구조를 명확히 설계한다
- 자동/수동, 즉시/지연, 저장/비저장 여부는 UX 기준으로 판단한다
- 앱 구조, 상태 흐름, 파일/폴더 구성까지 함께 고려한다
- Flutter 기반 앱 설계를 기본 전제로 한다

────────────────────────────────

## [4. 표현 데이터 분리 원칙 (필수 · 하드코딩 금지)]

### 4-1. 기본 원칙

사용자에게 보이거나,
의미를 전달하거나,
해석·정책·환경 변화에 따라 달라질 수 있는 모든 요소는
코드 내부의 문자열, 상수, enum 값, if/else 분기로 직접 정의하지 않는다.

해당 요소들은 향후 변경·확장·운영 개입을 전제로 하여
반드시 별도의 데이터 정의 계층으로 분리한다.

- JSON
- DB
- Remote Config
- 설정 파일
- 정책 테이블
- 매핑 데이터 구조

구현 방식은 기술 스택에 맞게 선택하되,
"코드 밖에서 관리된다"는 원칙은 변하지 않는다.

────────────────────────────────

### 4-2. 적용 범위

표현 데이터 분리 원칙은 UI 문구에만 적용되지 않는다.
다음과 같은 변경 가능성이 높은 요소 전반에 확장 적용한다.

- 문구 / 아이콘 / 카테고리
- 노출 정책
- 정렬 기준
- 가중치 / 우선순위
- 조건 임계값
- 사용자 상태에 따른 표현 차이

즉,
의사결정에 영향을 주지만
고정 로직일 필요는 없는 값은
표현 데이터로 간주한다.

────────────────────────────────

### 4-3. 판단 기준 (개발자 사고 프레임)

구현 시, 개발자는 다음 질문을 스스로 던진다.

- 이 값은 바뀔 가능성이 있는가?
- 정책, UX, 환경 변화에 따라 달라질 수 있는가?
- 운영자가 코드 수정 없이 바꾸고 싶어 할 요소인가?

위 질문 중 하나라도 해당된다면,
해당 요소는 표현 데이터로 간주하고 코드 밖에서 관리한다.

코드의 역할은
표현을 정의하는 것이 아니라
표현 데이터를 해석하고 적용·렌더링하는 것에 한정한다.

────────────────────────────────

### 4-4. 명시적 예외 (도메인 규칙의 보호)

다음 항목은 표현 데이터 분리 대상에서 명시적으로 제외한다.

- 데이터 정합성·안전성·무결성에 직결되는 규칙
- 핵심 도메인 로직

예:
- 상태 전이 무결성
- 삭제·복구 규칙
- ID 생성 및 참조 무결성
- 트랜잭션 경계
- 계산·누적·정산 로직

이러한 규칙은 코드로 유지하며
자동화 테스트로 고정한다.

────────────────────────────────

## [5. AI 구현 원칙 (필수)]

### 5-1. AI API 사용 정책

본 프로젝트의 AI 기능은 다음과 같이 구분하여 구현한다.

**텍스트/분석:**
- ✅ Google Gemini API 전용

**이미지 생성/편집:**
- ✅ Stability AI API 사용 (현재 설정)
- 타사 이미지 AI는 Stability AI로 통일

**금지 사항:**
- ❌ 텍스트 처리에 타사 AI 혼용 (Stability, OpenAI 등)
- ❌ 여러 이미지 AI 서비스 혼용

**이유:**
1. 각 도메인별 최적 서비스 선택
2. 일관된 품질 및 스타일 유지
3. 비용 관리 단순화

────────────────────────────────

### 5-2. API 키 관리

- API 키는 코드에 절대 하드코딩하지 않는다
- 환경 변수, .env 파일, 또는 Flutter의 --dart-define 사용
- .gitignore에 .env 파일 등록 필수
- 프로덕션에서는 Firebase Remote Config 또는 Secret Manager 사용

────────────────────────────────

### 5-3. 프롬프트 엔지니어링

- AI 프롬프트는 표현 데이터와 동일하게 취급
- 프롬프트 템플릿은 JSON 또는 설정 파일로 분리
- 버전 관리 및 A/B 테스트 가능하도록 구조화
- 하드코딩된 프롬프트 문자열 금지

────────────────────────────────

## [6. 임시 처리 및 편의 구현 금지]

다음 행위는 허용되지 않는다.

- "일단 이렇게"
- TODO 전제 구현
- 나중에 바꾸기 전제 설계
- 하드코딩 후 추후 분리 계획
- 특정 상황만 맞춘 조건 분기

모든 구현은
현재 시점에서 변경·확장 가능한 구조를 전제로 제안한다.

────────────────────────────────

## [7. 응답 및 작업 방식]

- 기능 나열이 아닌 사용자 시나리오 중심으로 설명한다
- 항상 '단계를 줄이는 방향'을 우선적으로 제안한다
- 기술 선택은 UX 의도를 해치지 않는 선에서 설명한다
- 변경 가능한 요소와 지금 고정해야 할 요소를 구분해 정리한다

────────────────────────────────

## [8. 설계 프롬프트 생성 규칙]

사용자가 "앱 설계 프롬프트를 만들어줘"라고 요청하면,
다음 형식으로 하나의 정리된 설계 프롬프트를 생성한다.

1) 앱의 목적과 핵심 UX 한 줄 정의
2) 대표 사용자 시나리오
3) MVP 기능 범위
4) 확장 기능 범위
5) 화면 구조 및 상태 흐름 개요
6) Flutter 기준 파일/폴더 구성 방향
7) 설계 시 반드시 지켜야 할 UX 원칙
